## Canvas Traceability Fields — Methodology

Added 2026-07-18 to close spec-auditor Phase 0.7 BLOCKER B-1 (all six fields were missing on every row). Derivation rule per field, applied uniformly across all 25 rows — no Canvas link was invented beyond what the PRD's per-REQ `Canvas:` field or the Canvas document itself already states:

- **canvas-link** — the same Canvas Item ID(s) already linked to that REQ in the PRD's per-REQ `Canvas:` field (identical content to the existing `Canvas Item ID` column, wrapped as a link into the Canvas doc). No new Canvas IDs were introduced.
- **canvas-problem** — [CAN-001](../canvas/shark-engine-air-release-1.canvas.md), uniformly. CAN-001 is the single documented Canvas problem statement; every requirement in this PRD exists to help resolve it, so this reference does not vary per row.
- **canvas-target-user** — [CAN-002](../canvas/shark-engine-air-release-1.canvas.md) (primary: Minecraft-Spieler, die Fahrzeuge bauen/steuern) by default. [CAN-003](../canvas/shark-engine-air-release-1.canvas.md) (sekundär: passive Copiloten) is added only where the requirement's own text explicitly concerns the copilot/passenger role (REQ-008, REQ-009, REQ-010, REQ-011, REQ-017) — not asserted for rows where copilots aren't mentioned.
- **canvas-value-claim** — [CAN-004](../canvas/shark-engine-air-release-1.canvas.md), uniformly. CAN-004 is the single Canvas value promise; each requirement is one facet of delivering it.
- **canvas-success-signal** — [CAN-019](../canvas/shark-engine-air-release-1.canvas.md) for all P0 rows, since CAN-019 ("Alle P0-Anforderungen und Akzeptanzkriterien sind ... nachgewiesen") is literally scoped to P0. For the three P1 rows (REQ-018, REQ-019, REQ-020) and the one P2 row (REQ-023), the field still cites CAN-019 but appends an explicit qualifier that these are outside CAN-019's stated P0 scope, rather than silently overclaiming P0-gate status for P1/P2 work.
- **canvas-risk-status** — one of `aligned | value-risk | non-goal-violation | risk-introduced | blocked`, judged per row from the requirement's own already-linked `Risks` column read against Canvas non-goals CAN-012/CAN-013/CAN-014 and the Risks register in the PRD:
  - `aligned` — the requirement's current (post-remediation) wording enforces or stays inside a Canvas non-goal/success boundary rather than threatening it: REQ-002 and REQ-023 explicitly keep LAND/WATER and Looping non-executable (CAN-012/CAN-013).
  - `value-risk` — the requirement carries an open (non-RESOLVED) risk in the PRD's Risk register that could degrade release value/quality if not carefully executed (RISK-003 desync, RISK-005 generalization pressure, RISK-006 particle performance, RISK-001/002 schedule/baseline exposure). REQ-025's row is `value-risk`, not `aligned`, specifically because — per the reworded RISK-002 entry below — REQ-025's Day-0 gate fully closes RISK-001 but only *partially* mitigates RISK-002's during-window integration-error content; a residual risk stays attached to that row (see docs/intake/PHASE-0.16-COUNCIL-CHALLENGE.md for the council reasoning behind REQ-025). REQ-024 is `value-risk` for the same underlying reason, corrected 2026-07-18 by Plumbline Watcher advisory: it is the single, un-decomposed end-of-release evidence gate that carries RISK-002's residual during-window exposure by design, since the user explicitly declined the council's incremental per-REQ gating alternative that could have reduced it — not `aligned`, which would understate that this checkpoint is a single roll of the dice at day ~12.
  - `risk-introduced` — the requirement's own implementation mechanically creates a new duplication/data-loss risk if built wrong, independent of any pre-existing risk: REQ-003, REQ-012, REQ-013, REQ-014, REQ-021 (all tied to RISK-004, non-atomic edit/reassembly).
  - `non-goal-violation` — not used for any row in this finalized PRD: REQ-022 (the one requirement that could have crossed into CAN-012/CAN-011 non-goal territory) was demoted by the Phase 0.16 council to extract only AIR's actually-called seams before this PRD was finalized (see docs/intake/PHASE-0.16-COUNCIL-CHALLENGE.md); its row is `value-risk` (residual RISK-005 generalization pressure, still tracked as ASSUMPTION/unresolved), not a violation.
  - `blocked` — not used for any row: all five Open Questions are RESOLVED and REQ-025's Day-0 gate is a workflow sequencing rule, not a per-requirement Canvas-alignment blocker.

Every `REQ-*` in `docs/prd/shark-engine-air-release-1.prd.md` must appear exactly once or more in this matrix and link to at least one `VIS-*`, `CAN-*`, `AC-*` and `EV-*` item.
